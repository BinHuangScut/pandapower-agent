# 让电网分析“会说人话”：pandapower-agent 是怎么工作的

在很多团队里，电网仿真能力并不缺，缺的是“让更多人用起来”的最后一公里：

- 工具能力强，但门槛高
- 分析流程能跑，但不够连续
- 结果有数据，但不够易读

`pandapower-agent` 的目标，就是把这段距离补上。
它把自然语言交互、结构化工具调用和 `pandapower` 数值引擎组合在一起，让“提出问题 -> 执行分析 -> 给出结论 -> 导出结果”变成一个连贯流程。

## 一句话介绍

`pandapower-agent` 是一个面向电网分析的 CLI 智能体。
你可以直接用中文或英文下达任务，它会自动调用合适的分析工具，返回可读结论，并支持场景对比、回滚和 JSON 导出。

## 它解决了什么问题

### 1. 把“会算”变成“会用”

过去常见流程是：写脚本、调参数、跑计算、再手工解释。
现在可以直接说：

- “运行交流潮流并总结风险”
- “做 N-1 筛查，返回最严重前 10 个故障”
- “在 3 号母线加 5MW 负荷后重算并对比”

系统会把自然语言转换为结构化工具调用，自动完成执行链路。

### 2. 把一次性分析变成连续实验

项目内置场景机制，可以自然支持：

- 保存场景（`save`）
- 加载场景（`load`）
- 对比场景（`compare`）
- 回滚上一步（`undo`）

这对“调参-重算-比较-复盘”的工程工作流很关键。

### 3. 把结果从“终端输出”升级为“可复用资产”

除可读结论外，系统支持导出：

- `summary`：关键指标摘要
- `results`：完整结构化结果

并提供进程外缓存兜底，避免结果丢失后无法复盘。

## 技术上是怎么做到的

整体架构很清晰：**LLM 负责调度，仿真引擎负责真值**。

1. CLI 接收用户指令（`run/chat`）
2. Agent 按 provider 使用 OpenAI SDK 发起工具调用（OpenAI Responses / Google Chat Completions）
3. 工具层按白名单执行具体分析能力
4. `pandapower` 完成数值计算
5. 结果以表格 + 结构化摘要返回给用户

这套设计的核心是“可控”：

- 工具有 schema，参数有校验
- 改网操作有快照，可回滚
- 错误有统一格式，方便排查

## 已覆盖的核心能力

- 网络发现与切换（内置算例）
- AC/DC/三相潮流
- 短路分析
- 拓扑分析（含失供母线识别）
- N-1 故障筛查
- 网络诊断
- 元件编辑（负荷、静态电源、线路、变压器等）
- OPF 与状态估计

这意味着它既能做“快速评估”，也能做“状态化实验”。

## 对团队协作有什么价值

### 对电力工程师

- 减少重复脚本操作
- 更快完成方案对比
- 保留结构化结果便于复核

### 对算法/平台团队

- 工具接口标准化，便于扩展
- 新能力可按“参数模型 + handler + 注册”快速接入
- 测试覆盖关键控制流，降低迭代风险

### 对业务和管理侧

- 可直接阅读的自然语言结论
- 有依据的关键指标摘要
- 易演示、易复现、易归档

## 为什么这类产品现在值得做

在电力数字化场景里，大家已经有了大量算法和模型积累。
下一步竞争点，越来越是“谁能更快把能力变成可交付流程”。

`pandapower-agent` 的意义就在这里：
不是替代专业分析，而是降低专业能力被使用和复用的摩擦成本。

## 结语

`pandapower-agent` 不是一个只会聊天的 AI 壳子。
它是一个把自然语言入口、可控工具链和可信仿真内核连接起来的工程系统。

如果你希望电网分析从“专家脚本”走向“团队能力”，这就是一个可直接落地的起点。
